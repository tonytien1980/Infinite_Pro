# 03_system_overview.md

> 文件狀態：Baseline v1.1（補強版，可作為後續文件與開專案依據）
>
> 本文件用於用白話方式說明 AI Advisory OS 的完整系統輪廓、MVP 在整體系統中的位置、Ontology 的角色、知識來源與知識庫的關係、未來模組的擴充方式，以及為什麼現階段不需要一次把完整宇宙都做完。

---

## 1. 這份文件是拿來幹嘛的

這份文件不是在講資料表，也不是在講技術細節。

它只回答一件最重要的事：

> **AI Advisory OS 整體到底會長成什麼樣子，而現在的 MVP 又是在做其中哪一塊？**

如果這件事沒有先看懂，後面不管是 Ontology、Agent 架構還是系統設計，都會變得很抽象。

所以這份文件的目的，是先幫你建立整體畫面。

---

## 2. 先講一句最白話的定義

AI Advisory OS 可以先把它想成：

> **一個能吃進問題與背景資料、由主持人 AI 調用不同專業 AI 一起分析，最後輸出報告、建議與行動項，並且之後還能一直加新能力模組的智能工作平台。**

它不是單一聊天機器人。

它也不是只會做摘要的知識庫。

它更像是一個：
- 有中樞
- 有流程
- 有共享世界模型
- 有不同專業能力
- 有結果沉澱
- 有後續擴充空間

的系統。

---

## 3. 完整系統可以先想成 6 個區塊

### 3.1 使用者操作區

這是你實際看到的介面。

你可以在這裡：
- 輸入問題
- 上傳文件
- 指定任務
- 查看分析結果
- 保存歷史任務
- 回頭看過去產出

這一層就像系統的入口大廳。

---

### 3.2 主持人 Agent 中樞

這是整個系統的大腦。

主持人 Agent 不只是回答問題，它負責：
- 先理解你丟進來的是什麼任務
- 判斷該怎麼處理
- 決定要不要叫其他 Agent 出來幫忙
- 控制分析順序
- 避免討論發散
- 把結果收斂成你真的能用的東西

你可以把它想成：

> **系統裡的總指揮 / 主持人 / 編排器。**

沒有這一層，系統就很容易退化成「很多 AI 各講各的」。

---

### 3.3 資料來源與知識入口區

這一層是你剛剛特別在意、也非常關鍵的部分。

它負責接住：
- 上傳檔案
- 任務背景資料
- 歷史專案文件
- 未來可能接入的雲端來源（例如 Google Drive 類型來源）

這一層的角色不是直接做分析，而是：

> **把真實世界的資料帶進系統，成為後續知識與分析的原料入口。**

如果沒有這一層，系統再聰明也很容易變成空想型 AI。

---

### 3.4 Ontology 本體論核心區

這是這套系統最核心的地方。

它的作用不是讓系統背很多定義，而是讓系統知道：
- 現在在處理的是什麼任務
- 背景資料是什麼
- 分析的對象是什麼
- 目標是什麼
- 風險是什麼
- 建議是什麼
- 行動項是什麼

也就是說：

> **Ontology 不是裝飾，而是讓整個系統活在同一個世界裡的底座。**

主持人 Agent 和其他 Agent，不是各自幻想問題，而是共享這個世界模型。

---

### 3.5 Agent 能力區

這一層就是各種可被調用的 AI 能力。

例如未來可能有：
- 策略分析 Agent
- 營運診斷 Agent
- 行銷洞察 Agent
- 風險挑戰 Agent
- 合約審閱 Agent
- 研究整理 Agent
- 提案重組 Agent

這些 Agent 不是系統本身，
而是建立在 Ontology 之上，由主持人 Agent 視情況調用的能力模組。

所以你未來要擴充的東西，很多都會發生在這一層。

---

### 3.6 結果與歷史沉澱區

這一層負責把每次工作成果留下來。

不是只留下聊天紀錄，
而是留下：
- 報告
- 建議
- 行動項
- 使用過的背景資料
- 任務歷史
- 可回顧的分析結果

這一層很重要，因為你的系統如果沒有沉澱，
每次就只是一次性的 AI 對話。

而你真正想做的是一個會越用越有價值的系統。

---

## 4. 檔案、知識庫、Ontology 之間的關係

這三個概念很像，但不是同一件事。

### 4.1 檔案 / 雲端來源
例如：
- 上傳的 PDF / DOCX / 簡報
- 合約檔
- 客戶需求文件
- 研究資料
- 未來可能接入的 Google Drive / Notion 類型來源

這一層是**資料來源**。

### 4.2 知識庫
當檔案與資料不只是進來一次，而是能被整理、引用、重用時，
它們就開始形成**知識庫**。

也就是說，知識庫比較像：
- 可查詢
- 可引用
- 可跨任務重用
- 可逐步累積的資料層

### 4.3 Ontology
Ontology 不是把檔案存起來，
而是把檔案與背景內容映射成系統能理解的工作物件，例如：
- Task
- Evidence
- Insight
- Risk
- Recommendation
- ActionItem

所以更精確的說法是：

> **檔案 / 雲端來源是原料，知識庫是可重用記憶層，Ontology 是把這些原料轉成工作世界物件的方式。**

---

## 5. 用一句圖像化方式理解整體系統

你可以把整個 AI Advisory OS 想成：

- **前台**：你輸入任務、上傳文件、看結果
- **中控室**：主持人 Agent 決定怎麼跑流程
- **知識入口**：上傳檔案與未來雲端來源把資料帶進來
- **世界模型**：Ontology 提供所有 Agent 的共通理解
- **專業團隊**：各種 Agent 負責不同視角或不同單點能力
- **成果倉庫**：保存報告、建議、行動項與歷史任務

這樣你就不會把它誤會成「很多聊天機器人湊在一起」。

---

## 6. MVP 在整體系統中的位置

這一點最重要。

MVP 不是整個系統的全部，
它只是這個系統的第一個可運作核心。

你可以把它想成：

> **先蓋出整座城市的市中心，而不是一次把整座城市蓋完。**

V1 / MVP 現在要做的是：

### 6.1 先把這幾件事做起來
- 有任務輸入
- 有檔案上傳與背景引用
- 有主持人 Agent
- 有 3~5 個核心 Agent
- 有最小可行 Ontology
- 有結構化報告與行動項輸出
- 有任務歷史保存

### 6.2 暫時不做這些
- 不做完整 Agent Market
- 不做所有產業的完整本體論
- 不做完整企業權限系統
- 不做全自動高風險執行
- 不做所有單點工具的全面替代
- 不做完整雲端知識平台

也就是說，MVP 不是小一點的最終產品，
而是：

> **先把未來整個平台最重要的核心循環跑通。**

---

## 7. 這個核心循環到底是什麼

MVP 最重要的不是功能列表，
而是先把一條完整價值鏈跑通。

我建議你把它理解成下面這條鏈：

### 7.1 使用者提出任務
例如：
- 幫我分析這個客戶需求
- 幫我整理這份研究
- 幫我審閱這份合約
- 幫我從多角度判斷這個商業問題

### 7.2 使用者提供背景
例如：
- 上傳文件
- 貼上需求
- 提供說明
- 補充限制條件

### 7.3 系統整理知識來源
也就是：
- 把檔案、文字說明、背景資料整理成可引用來源
- 準備給 Ontology 與 Agent 使用

### 7.4 主持人 Agent 判斷任務模式
它要先決定：
- 這是複雜問題，要啟動多 Agent 收斂？
- 還是這是單點 Specialist 任務？

### 7.5 系統載入 Ontology 與需要的 Agent
也就是：
- 把這次任務放進系統理解的世界裡
- 把需要的角色與能力調出來

### 7.6 系統進行分析
這時候：
- 主持人 Agent 管流程
- 不同 Agent 從不同角度提出分析
- 系統整理風險、洞察、方案與建議

### 7.7 系統收斂輸出
最後不是只回你一段話，
而是輸出：
- 問題定義
- 重點分析
- 風險
- 建議
- 行動項
- 任務成果

### 7.8 系統保存歷史
這樣你之後可以：
- 回顧
- 修改
- 延續任務
- 沉澱方法論
- 累積專案知識

這條鏈一旦跑通，整個平台就活了。

---

## 8. 為什麼核心是 Ontology，但現在不用一次做完整宇宙

### 8.1 對，Ontology 是核心
沒有 Ontology，這套系統很容易變成：
- 一堆 prompt
- 一堆聊天流程
- 一堆沒有共同語境的 AI 角色

所以本體論一定要先做。

### 8.2 但現在不用一次做完整宇宙
因為你真正需要先做的是：
- 先把本體論的**核心骨架**做出來
- 先讓它能支撐 MVP 的工作流
- 先驗證它在真實任務裡真的有價值

而不是一開始就去定：
- 所有產業
- 所有職位
- 所有模組
- 所有規則

### 8.3 你現在該追求的完整，不是「範圍完整」
而是：
- **方法完整**
- **主流程完整**
- **擴充邏輯完整**

也就是說：

> **不是先把宇宙全部畫完，而是先把宇宙的運作規則和第一個核心區域做出來。**

---

## 9. 那未來的擴充模組是怎麼長出來的

未來的模組，不會是另外長出一套完全不同的系統。

它們應該是：

> **建立在同一套 Ontology + 主持人流程 + Agent 能力調用架構之上的擴充能力。**

例如：

### 9.1 單點模組
- 合約審閱模組
- 標案拆解模組
- 市調整理模組
- 設計前期研究模組

這些模組比較像是：
- 特定任務流程
- 特定 Specialist Agent 組合
- 特定輸出模板

### 9.2 複合型模組
- 公關提案模組
- 教育內容規劃模組
- 顧問診斷模組
- 品牌策略模組

這些模組會比較依賴：
- 主持人 Agent
- 多 Agent 協作
- 更完整的背景引用
- 更複雜的收斂方式

### 9.3 未來平台級擴充
- 更多 Agent 類型
- 更多產業包
- 更多工具串接
- 團隊協作
- Agent Registry / Market
- 雲端來源同步
- 專案級知識庫

所以模組不是亂加，
而是沿著同一個骨架長出來。

---

## 10. 現在什麼一定要先做，什麼可以後做

### 10.1 現在一定要先做的底座
這些是未來所有模組都會依賴的：
- 任務入口
- 檔案上傳與背景引用
- Host Agent
- 基本多 Agent 流程
- 最小可行 Ontology
- 結構化輸出
- 歷史保存
- 基礎模型切換能力

### 10.2 之後可以慢慢加的模組與擴充
- 更多 Specialist 任務
- 更多產業包
- 更細的本體論
- 團隊工作區
- 權限與治理
- 外部 Agent 市場
- 更進階的執行追蹤
- 雲端來源連接器
- 專案級知識重用

也就是說：

> **現在先做底座，不是先做全部。**

---

## 11. 你現在真正需要確認的是什麼

不是確認每一個本體名詞有沒有百分之百正確，
而是先確認下面這件事：

> **你想做的系統，是不是就是這種「一個有本體核心、有主持中樞、有可擴充 Agent 能力、有知識入口、有結果沉澱、未來可長出各種模組的智能工作平台」？**

如果這個總圖你認同，
那後面的 Ontology 文件、Agent 文件、System Architecture 文件就都只是把這張總圖一層一層具體化。

---

## 12. 這份總覽如何幫助你後面不迷路

### 12.1 幫你確認自己不是在做一堆零散功能
而是在做一個可擴張的底座。

### 12.2 幫你確認 MVP 的角色
MVP 不是先做一個很小很弱的工具，
而是先做整體系統的第一個可運作核心。

### 12.3 幫你確認 Ontology 在哪裡
Ontology 不是獨立存在的理論層，
而是整個系統的共享世界模型。

### 12.4 幫你確認知識入口在哪裡
上傳檔案與未來雲端來源，不是附加功能，
而是讓 Ontology 真的接住真實案子的來源層。

### 12.5 幫你確認未來模組怎麼掛
未來不是每來一個需求就重做一套系統，
而是沿著同一個底座擴充。

---

## 13. 結語

AI Advisory OS 的正確理解方式，不是把它看成一個「會回答問題的 AI」，
而是把它看成一個：

- 以 Ontology 為核心
- 以 Host Agent 為中樞
- 以可擴充 Agent 能力為工作手腳
- 以檔案 / 雲端來源作為知識入口
- 以結構化輸出與歷史沉澱為價值累積
- 以未來模組與場景擴充為長期方向

的智能工作平台。

而 MVP 的任務，不是一次把這個平台全部做完，
而是先把這個平台最核心、最有價值、最能驗證方向的那一塊真正做起來。

> **先把底座做對，未來宇宙才長得起來。**
