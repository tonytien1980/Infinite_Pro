## 2. 一句話定義

Infinite Pro 是一套**以 Ontology 為底層世界模型、以多 Agent 協作推理為方法、以主持人 Agent 收斂與治理為核心、並可自由切換 LLM API 的獨立顧問型智能系統**。

它不是單一聊天機器人，也不是單一模型的包裝層，而是一個能夠針對複雜商業問題進行多視角分析、討論、辯證、收斂，並輸出報告、建議與行動方案的顧問作業系統。

---

## 3. 專案背景與動機

在真實世界的顧問工作、策略規劃與企業決策場景中，問題通常具備以下特徵：

1. **跨部門**：同一個問題往往同時牽涉財務、營運、行銷、產品、法務、HR 等觀點。
2. **跨產業**：顧問面對的客戶可能來自 SaaS、製造、遊戲、金融、零售、醫療等不同領域。
3. **跨資料來源**：判斷依據通常散落於報表、文件、資料庫、會議紀錄、Excel、CRM、ERP、郵件與外部市場資訊之中。
4. **高不確定性**：很多重要問題並沒有單一正確答案，而需要在不完整資訊下進行推理與取捨。
5. **需要收斂**：真正有價值的不是討論本身，而是能否形成清楚的決策建議、優先順序與執行清單。

傳統顧問工作高度依賴人力、會議與經驗，效率低、可重複性差、知識難沉澱。傳統 AI 助理雖然能做摘要與問答，但往往缺少共享世界模型、角色分工、辯證機制與行動收斂能力。

因此，本專案的核心動機是打造一套：

- 能理解企業/專案/產業運作結構的系統
- 能模擬多種專業角色共同討論的系統
- 能由主持人 Agent 將複雜討論收斂成決策輸出的系統
- 能不綁定單一 LLM 供應商、具備長期擴充性的系統

## 4. 產品願景（Vision）

打造一個**可作為企業、顧問團隊與決策者之智能協作中樞**的系統，讓 AI 不只是回答問題，而是真正能夠：

- 理解複雜商業情境
- 調用多種專業視角
- 在共享 Ontology 上進行推理與辯證
- 收斂為有依據的建議與行動方案
- 持續沉澱知識、流程與判斷邏輯

長期願景不是做出「最會聊天的 AI」，而是做出一套：

**能夠陪伴顧問與決策者處理真實問題的智能顧問作業系統。**

## 5. 產品定位（Positioning）

本產品的定位不是通用聊天機器人，也不是單純報告生成器，而是：

### 5.1 核心定位

**一個以 Ontology 為核心、具備高度擴充性、可動態載入不同 Agent 能力、並由主持人 Agent 統一編排與收斂的智能工作平台。**

它的本質不是服務某一個固定產業、某一種固定職能，或某一種固定規模的組織，而是：

- 以 **Ontology** 作為共享世界模型
- 以 **Agent 可擴充能力** 作為專業與任務的承載方式
- 以 **主持人 Agent** 作為流程治理、邏輯收斂與輸出品質控制核心
- 以 **LLM Provider 可切換架構** 保持模型供應商中立
- 以 **Structured Report and Action Output** 作為實際工作成果輸出形式
- 以 **外部工具對接能力** 支撐後續執行、追蹤與工作流延伸

### 5.2 與一般 AI 產品的差異

| 類型 | 核心特徵 | 本產品差異 |
|---|---|---|
| 聊天型 AI | 單輪/多輪問答 | 本產品重視世界模型、角色編排、邏輯收斂與任務輸出 |
| 知識庫 AI | 文件檢索與摘要 | 本產品強調 Ontology、Agent 擴充、主持治理與行動導向輸出 |
| BI / Dashboard | 資料呈現與指標查看 | 本產品希望進一步做判斷、辯證、方案建議與後續追蹤 |
| Workflow 工具 | 固定流程執行 | 本產品適合高不確定性與高判斷密度場景，也可兼容單一任務型功能 |
| 單一 Agent 系統 | 一個 AI 負責全部 | 本產品採多角色、多能力、主持人收斂架構，並保有單一功能型 Agent 的使用彈性 |

### 5.3 產品初衷

本產品的核心精神是：

**無限進步、無限想像。**

也就是從產品名稱、架構、功能與平台思維上，都必須圍繞：

- 可持續擴充
- 可持續進化
- 可承載越來越多能力
- 可從單點工具走向完整系統
- 可從單一問題處理走向多角色、多步驟、多工具的複合型工作流

## 6. 目標使用者（Target Users）

本產品雖由顧問需求出發，但不應被侷限為「只有顧問能用」的系統。其本質更接近一套**可支援知識型工作、研究型工作、提案型工作、決策型工作、審閱型工作與執行追蹤型工作**的智能協作平台。

同時，本產品不應預設只有「複雜商業決策」才能使用；它也必須能支援**單一功能型任務**，例如：
- 合約生成
- 合約審閱
- 文件分析
- 研究整理
- 報告草稿生成
- 標案內容拆解
- 設計前期調研

### 6.1 第一層核心使用者

1. **各類顧問型工作者與顧問團隊**
   - 策略顧問
   - 管理顧問
   - 營運顧問
   - 行銷/品牌顧問
   - 財務顧問
   - 法務/合規相關顧問
   - 提案與商業分析人員

2. **需要大量研究、整理、提案、審閱與判斷的人員**
   - 公關公司團隊
   - 市場調查/研究公司
   - 補教/線上教育內容規劃團隊
   - 設計師與設計團隊（前期研究、競品、策略、概念收斂）
   - 標案/企劃/商務提案人員
   - 法務/合約處理/文件審閱型角色

3. **企業內部決策與高階幕僚角色**
   - CEO / Founder
   - COO / Strategy Lead
   - CFO / 財務長
   - CLO / General Counsel / 法務長
   - CMO / 行銷長
   - PMO / 營運主管
   - 財務/營運分析人員
   - 特別專案負責人

### 6.2 使用者規模

本產品在理論上可服務：
- 個人專業工作者
- 小型顧問或專案團隊
- 中小企業部門
- 中大型企業的策略/幕僚/專案單位

但從產品思維來看，與其先把所有適用對象列滿，不如先確認一個更重要的原則：

> **本產品的價值核心不在於預先定義所有可能的產業與角色，而在於能否透過 Ontology + Agent + 主持人編排，持續擴充出適合不同場景的能力組合。**

因此，「目標使用者」在本產品中，更應視為**可逐步擴展的適用範圍**，而不是一開始就完全固定的邊界。

## 7. 核心價值主張（Value Proposition）

本產品希望為使用者提供以下核心價值：

### 7.1 讓複雜問題不再只靠人力會議處理
透過多 Agent 協作與主持人編排，將原本需要大量會議、跨部門討論與顧問整理的工作，轉化為可重複、可追溯、可收斂的智能流程。

### 7.2 讓 AI 不只是講答案，而是講出推理結構
系統應能區分：
- 哪些是事實
- 哪些是推論
- 哪些是假設
- 哪些是建議
- 哪些資料仍不足

### 7.3 讓平台既能處理複雜決策，也能支援單一功能型任務
本產品不只應支援高複雜度、多角色、多步驟的商業問題，也要能因應需求快速切換為單一任務型能力，例如合約審閱、文件生成、研究整理等。

### 7.4 讓多專業視角在同一個世界中討論
不同 Agent 不該各說各話，而應建立在共享 Ontology 之上，對同一個商業世界進行不同視角分析。

### 7.5 讓知識、能力與方法論能夠沉澱與擴張
每次討論不只是一次性對話，而是能沉澱為：
- 決策紀錄
- 問題分類
- 報告模板
- 風險判斷模式
- 專業 Agent 能力包
- 公司/產業客製知識

### 7.6 讓平台具備長期擴充性
不綁定單一模型供應商，並允許未來透過 Agent Registry / 受控式 Agent Market 持續擴充專業能力、任務模組與場景組合。

## 8. 產品核心能力（Core Capabilities）

### 8.1 Ontology-Based Context Modeling
將組織、角色、部門、流程、KPI、風險、決策、任務、客戶、產品等核心物件建立為可被系統理解的世界模型。

### 8.2 Expandable Agent Capability System
系統應能依需求創造、載入、組合不同專業與任務能力的 Agent，而不被單一職位、單一產業或單一用途綁死。

### 8.3 Multi-Agent Reasoning
根據任務需求，調用多個專業視角 Agent 進行獨立分析、相互質疑、比較與辯證。

### 8.4 Host-Led Convergence
由主持人 Agent 負責任務拆解、選角、流程控制、矛盾收斂、結論整理與輸出治理。

### 8.5 Single-Task Specialist Execution
除了多 Agent 複合工作流，系統也應支援單一功能型 Specialist Agent 的快速調用，例如：合約審閱、提案整理、競品研究、文件重組等。

### 8.6 Model Routing and Provider Independence
系統內部不綁定單一 LLM，未來可依成本、速度、能力、任務型態選擇不同 API。

### 8.7 Structured Report and Action Output
輸出不只是一段對話，而是結構化的：
- 問題定義
- 事實整理
- 分析結論
- 方案比較
- 建議採納方向
- 後續行動項
- 執行與追蹤接口

### 8.8 External Tool Integration
平台應保留與外部工具、資料來源與工作流系統對接的能力，以支持更完整的落地應用。

## 9. 產品設計原則（Design Principles）

### 9.1 先有世界模型，再有對話
系統的核心不是 prompt，而是 Ontology。

### 9.2 先有能力擴充，再有場景擴充
不要先把產品綁死在某個產業、某種職位或某個單一用途，而是先確保系統具備承載不同能力模組的架構彈性。

### 9.3 先有角色視角，再有人格風格
Agent 的本質是「評估框架與決策視角」，不是單純模仿某個職稱說話。

### 9.4 討論必須可以收斂
所有討論都必須最終轉化為可閱讀、可比較、可執行的輸出，而不是無限延伸的對話。

### 9.5 AI 產出必須可追溯
每個關鍵結論都應能區分事實、推論、假設與不確定性。

### 9.6 系統必須同時支援複合型與單點型任務
既要能處理多角色、多步驟、多工具的複雜商業問題，也要能快速支援單一任務型功能使用。

### 9.7 高風險行動必須 human-in-the-loop
本系統可生成建議與判斷，但高風險決策與執行應保留人工核准機制。

### 9.8 系統必須具備平台化擴充能力
長期目標不是只做一個固定功能產品，而是形成可新增 Agent、知識包、產業包與工具連接器的平台。

## 10. 初步產品邊界（What It Is / What It Is Not）

### 10.1 本產品是
- 一套顧問型智能工作平台
- 一套多 Agent 協作推理系統
- 一套以 Ontology 為核心的決策支援系統
- 一套可延伸成 SaaS 與 Agent 生態的平台

### 10.2 本產品不是
- 單純聊天機器人
- 單純文件問答系統
- 傳統 Dashboard / BI 系統
- 完全自治、可無條件自動執行所有決策的系統
- 一開始就面向所有產業、所有場景的萬能產品

## 11. V1 建議切入方向（暫定）

為降低複雜度，V1 不應直接做成完整 Agent Market 與跨全產業平台，而應先聚焦在：

### 11.1 產品形態
**以顧問服務業為起點的內部主力工具型 MVP**

也就是：
- 先從對創建者自身幫助最大的場景開始
- 以顧問服務業作為第一個主要落地環境
- 用來提升研究、分析、提案、報告產出與決策支援效率
- 驗證主持人 Agent + 可擴充 Agent + Ontology + 報告/行動輸出的整體價值

### 11.2 V1 主產品體驗定位
**以複雜商業問題的多 Agent 收斂流程為主，單一功能型 Specialist 任務為輔。**

這代表 V1 的核心產品體驗應優先呈現：
- 使用者提出複雜問題、任務或情境
- 系統透過主持人 Agent 編排多個專業 Agent 進行分析、辯證與收斂
- 最終輸出具結構的報告、建議與行動清單

而單一功能型任務（如合約審閱、文件分析、研究整理等）在 V1 中仍可支援，但作為輔助能力存在，而非整個產品的主要敘事中心。

### 11.3 V1 主要能力
- 使用者輸入問題/任務
- 主持人 Agent 選擇少量核心 Agent
- 載入基礎 Ontology
- 進行多視角分析與收斂
- 輸出結構化報告與行動清單
- 支援部分單一功能型任務（如文件分析、合約審閱、研究整理）
- 保留後續任務追蹤、執行層與外部工具串接的擴充接口

### 11.4 V1 介面方向
**以個人工具台為主，但保留未來升級為小型團隊協作台的架構。**

這代表 V1 在體驗與設計上應優先滿足：
- 單一顧問或單一專業工作者高頻使用
- 快速輸入問題、啟動分析、查看過程與輸出結果
- 先降低多人協作與權限治理的設計複雜度

但在系統架構上，仍應預留未來升級空間，例如：
- Workspace / Project 概念
- 多使用者與角色權限
- 任務共享與歷史追蹤
- 小型團隊共同檢視與編修產出

### 11.5 V1 驗證邏輯
V1 的驗證不代表產品只屬於顧問服務業，而是：

- **先從顧問服務場景切入**，因為這是最貼近創建者需求、最容易獲得高頻真實回饋的場域
- 驗證成功後，再延伸至公關、教育、研究、設計前期、標案與其他知識型工作場景

### 11.6 V1 先不做
- 完整第三方 Agent 上架市場
- 全自動執行商業動作
- 全產業通用化的完整 Ontology
- 過度複雜的長期記憶與自主演化系統

## 12. 長期發展方向（Long-Term Direction）

本產品的長期商業模式，初期不是優先對外販售「平台帳號」，而是：

### 12.1 第一階段
**作為顧問服務的內部主力工具**

也就是：
- 系統本身先服務顧問團隊內部
- 幫助顧問更快完成研究、分析、提案、報告與決策支援
- 讓顧問服務的產能、品質與可複製性提升

### 12.2 第二階段
**形成半產品化的顧問工作台**

當內部驗證成熟後，可考慮：
- 讓部分客戶以工作台形式共同使用
- 將特定功能封裝成 SaaS 模組
- 將特定產業方法論轉化為產品化能力

### 12.3 第三階段
**平台化 / SaaS 化 / Agent 生態化**

若前兩階段驗證成功，未來可逐步進化為：

1. 多租戶 SaaS 平台
2. 支援多模型路由與成本優化
3. 內部 Agent Registry
4. 內部/受控式專家 Agent 上架機制
5. 產業 Ontology Packs
6. 公司客製化知識與流程層
7. 更完整的工作流、任務追蹤與核准流
8. 與企業內部工具/API 的深度整合
9. 決策品質評估與持續優化機制

## 13. 成功標準（Success Criteria）

本專案若要被視為有價值，至少應逐步達成以下成果：

### 13.1 使用者層面
- 使用者認為系統提供的分析比單一聊天 AI 更有深度與結構
- 使用者願意以其產出作為報告、討論或決策前置材料
- 使用者能感受到不同角色視角帶來的實質差異，而非重複發言

### 13.2 產品層面
- 系統可穩定調用多個 Agent 並完成收斂
- 輸出格式結構清楚，非單純自然語言堆疊
- 能夠逐步替換/切換不同模型供應商
- Ontology 與 Agent 設計可持續擴充

### 13.3 商業層面
- 具備形成 SaaS 產品的可能性
- 具備顧問型工作流的明確價值場景
- 具備未來 Agent 生態或行業模組化擴展的基礎

## 14. 已知風險（Known Risks）

1. **系統設計過大，難以在早期收斂**
2. **Ontology 若太抽象，會失去落地價值**
3. **多 Agent 討論容易變成高成本、低收斂**
4. **角色設計若流於人格化，會產生假專業感**
5. **若沒有良好的治理與權限設計，企業場景難以真正使用**
6. **若沒有明確 V1 切入場景，產品可能停留在概念層**

## 16. 結語

AI Advisory OS 的核心，不是讓很多 AI 角色一起聊天，而是建立一個：

- 有共享世界模型
- 有多視角專業推理
- 有主持人收斂治理
- 有供應商中立模型層
- 有未來平台化與市場化可能

的顧問型智能系統。

本文件的目的是先把這個方向講清楚，確保後續所有技術設計與文件撰寫，都圍繞同一個產品核心前進，而不是在實作過程中持續漂移。
