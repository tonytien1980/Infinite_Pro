# Dual-Layer Product Language System Design

日期：2026-04-13
狀態：draft

## 1. Purpose

這份設計文件要正式定義 Infinite Pro 下一輪的產品語言系統。

它要解決的不是單一按鈕要怎麼改字，
而是整個產品目前「看得懂，但不夠好懂」的問題。

這份文件要回答：

1. Infinite Pro 的正式第一層語言應服務誰
2. 哪些詞可以保留專業感，哪些詞不應直接出現在第一層
3. 怎麼讓產品同時維持專業、信任、穩定，又更容易上手
4. 哪些頁要更白話，哪些頁可以保留較高的專業濃度
5. 如何建立可持續的全站 copy 規則，而不是零碎改字

這份文件不回答：

- 每個句子最後要不要有句號
- 每個按鈕該幾個字
- 每個單頁最終實作細節

它的任務是先把語言系統定清楚，讓後續 copy pass 有一致標準。

## 2. Core Diagnosis

Infinite Pro 現在的文案問題，不是單純「還有些英文」。

真正的問題有四個：

### 2.1 第一層仍有太多系統語言

目前高流量頁仍然常看到這些詞：

- 工作面
- 寫回
- 治理
- 脈絡
- continuity
- decision brief
- canonical
- authority

這些詞不是完全不能用，
但不適合在第一層就直接要求一般使用者理解。

### 2.2 太多文字同時承擔太多責任

很多文案同時在做：

- 解釋這頁是什麼
- 解釋系統怎麼想
- 解釋為什麼這樣安排
- 指導下一步

結果不是更完整，
而是更難讀。

### 2.3 同一件事在不同地方重複說

目前常見情況是：

- hero 說一次
- guide 再說一次
- panel intro 又說一次

所以使用者感受到的是資訊量大，
而不是資訊真的比較完整。

### 2.4 專業感和可讀性還沒有分層

Infinite Pro 應該保留專業感，
但不是把專業感建立在抽象、系統化、半內部語言上。

真正成熟的專業產品應該是：

- 第一眼好懂
- 第二層才更專業
- 第三層才談系統內部結構

## 3. Recommended Direction

正式建議方向：

> Dual-Layer Product Language System

也就是：

- 第一層：全民可讀語言
- 第二層：顧問專業語言
- 第三層：系統內部語言

這不是把產品做得更幼稚，
而是把複雜度分層。

## 4. Language Layers

### 4.1 Layer 1: 全民可讀語言

這一層用在：

- header / nav
- hero
- 第一屏標題
- 主要 CTA
- 首屏段落導引
- 空狀態
- 成功 / 失敗 / loading 文案

這一層只回答：

- 我現在在哪裡
- 這裡能幫我做什麼
- 我下一步按哪裡

語氣要求：

- 白話
- 短句
- 動作導向
- 不假設使用者理解產品內部詞彙

### 4.2 Layer 2: 顧問專業語言

這一層用在：

- 第二層 disclosure
- detail section intro
- 較完整的結果解釋
- 交付摘要 / 建議 / 風險 / 行動
- 較成熟的案件與分析說明

這一層可以保留：

- 判斷
- 證據
- 限制
- 交付
- 風險
- 背景
- 研究
- 檢查點
- 推進狀態

但仍然不能變成工程師語言。

### 4.3 Layer 3: 系統內部語言

這一層只留在：

- advanced settings
- admin / management detail
- debug / technical diagnostics
- docs / architecture files

這一層才允許較直接出現：

- canonical
- authority
- provenance
- work slice
- governance
- writeback
- runtime
- fallback

這些詞原則上不直接進第一層。

## 5. First-Layer Rules

正式規則如下：

1. 第一層不用內部系統詞解釋頁面
2. 第一層按鈕必須像動作，不像概念
3. 第一層每段文字最多只傳一個主要意思
4. 第一層不要用長句串三個以上概念
5. 第一層不直接用英文術語，除非它是正式品牌名或不可替代名詞
6. 第一層若必須談系統，先翻成使用者聽得懂的語言

## 6. First-Layer Avoid List

以下詞彙原則上不直接放在第一層：

- 工作面
- 寫回
- 治理
- 脈絡
- continuity
- decision brief
- runtime
- fallback
- canonical
- authority
- provenance
- work slice
- progression

不是代表它們永遠不能出現，
而是不能當第一眼就要求使用者理解的語言。

## 7. Recommended Replacements

以下是正式替代方向：

| 目前常見說法 | 第一層建議說法 |
| --- | --- |
| 工作面 | 頁面 / 工作台 |
| 決策工作面 | 分析工作台 |
| 交付物工作面 | 結果與報告 |
| 工作紀錄 | 分析項目 / 這次分析 |
| 先補來源與證據 | 先補資料 |
| 主線 | 現在重點 |
| 最大 blocker | 目前卡住原因 |
| 寫回 | 同步回案件 |
| 治理 | 系統檢查 / 系統狀態 |
| 脈絡 | 背景資訊 |
| continuity | 延續狀態 / 後續狀態 |
| decision brief | 判斷摘要 |
| evidence chain | 證據支撐 |
| deliverable | 結果 / 報告 / 正式結果 |

注意：

- `案件`
- `交付物`
- `證據`

這三個詞已逐漸成為產品穩定語意，不建議粗暴全部替換。

## 8. Page-Level Guidance

### 8.1 `overview`

這頁應最白話。

它應像：

- 現在先做什麼
- 回哪裡繼續
- 要不要新建

不應像：

- 系統治理總覽
- 顧問成熟度報告
- 平台營運監控面板

### 8.2 `task`

這頁應第二白話。

因為它是最像「我要開始做事」的頁。

第一層應優先像：

- 這次分析在看什麼
- 現在先做什麼
- 做完去哪裡

不要太快掉進：

- 專家流程
- 代理路徑
- decision brief
- 系統研究主線

### 8.3 `evidence`

這頁應第三白話。

第一層應像：

- 現在缺什麼
- 要補什麼
- 補完回哪裡

不應第一眼就像：

- 證據缺口治理
- 研究來源脈絡整理
- 支撐鏈技術面板

### 8.4 `deliverable`

這頁保留最多專業感，
但第一層仍應像一份成熟報告工作台，
不是系統資訊牆。

第一層應優先像：

- 這是什麼版本
- 現在該發布、整理，還是回看
- 這份結果目前能不能直接用

### 8.5 `matter`

這頁可以比前面四頁稍微專業一點，
但第一層仍應先像案件主控台，
不是內部世界模型頁。

## 9. Management Surfaces

`agents / packs / settings / history` 可保留較高專業度，
但仍應遵守：

- 第一層標題要可讀
- 主要按鈕要像動作
- 不要把系統內部詞直接當導航名

這些頁比核心工作頁可以更專業，
但不應更難懂。

## 10. Copy QA Rules

後續 copy pass 必須逐頁檢查：

1. 第一層是否能讓一般使用者在 3 到 5 秒內理解
2. 第一層是否出現不必要的系統詞
3. 同一件事是否被不同區塊重複說
4. CTA 是否是動作語言
5. 長句是否能拆成一句話 + 2 到 3 個重點
6. 頁面是否仍維持專業、信任、穩定，而不是變成幼稚教學文案

## 11. Non-Goals

這一輪不追求：

- 全產品完全零英文
- 全部用字都改成口語聊天語氣
- 把顧問專業感打掉
- 一次改完整個產品所有深層 detail 文案

這一輪真正追求的是：

- 建立穩定的第一層語言規則
- 先讓高流量頁更容易懂
- 讓整個產品語氣更一致

## 12. Execution Priority

正式建議順序：

1. `overview`
2. `task`
3. `evidence`
4. `deliverable`
5. `matter`
6. `agents / packs / settings / history`

這個順序的原因是：

- 先處理第一次使用者最容易看到的頁
- 再處理最容易讓人迷路的頁
- 最後才處理管理型 surface

## 13. Acceptance Standard

這份語言系統完成後，應至少成立：

1. 第一層高流量頁不再要求使用者先理解系統內部詞彙
2. 第一層文案整體更短、更直接、更可行動
3. 第二層仍保留 Infinite Pro 作為專業顧問工作台的成熟感
4. 全站語氣更一致，不再像不同頁面在說不同產品
5. docs / code / QA evidence 之間的語言規則對得起來

## 14. Recommendation

正式建議採用：

> 第一層全民可讀，第二層顧問專業，第三層系統內部

Infinite Pro 不應被做成：

- AI hype shell
- 工程師介面
- 教學平台
- 只有高階顧問看得懂的工作台

它應該是：

- 一般人第一眼也知道怎麼走
- 顧問使用時仍然覺得專業
- 真正把複雜度藏到第二層和第三層的產品
