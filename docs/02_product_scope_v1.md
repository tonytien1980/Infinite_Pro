# 02_product_scope_v1.md

> 文件狀態：Baseline v1.1（補強版，可作為後續文件與開專案依據）
>
> 本文件用於定義 AI Advisory OS 第一版（V1 / MVP）的產品範圍、主要目標、核心功能、非目標、驗證指標與邊界，避免專案在早期因願景過大而失焦。

---

## 1. 文件目的

本文件的目的，是回答以下問題：

1. **AI Advisory OS 的第一版到底要做什麼？**
2. **第一版最重要的核心價值是什麼？**
3. **哪些功能可以先不做？**
4. **如何避免一開始就掉進過度平台化、過度工程化的陷阱？**
5. **第一版要如何作為未來平台化與 SaaS 化的基礎？**

本文件應作為後續系統架構、Ontology 設計、Agent 設計、介面規劃與 Codex 實作的重要邊界文件。

---

## 2. V1 產品定位

AI Advisory OS 的 V1，不是最終平台，也不是完整市場化產品，而是：

> **一個以顧問服務業內部使用為起點、以複雜商業問題收斂為主體、以個人工具台為主要體驗的內部主力工具型 MVP。**

V1 的任務不是證明「這個平台什麼都能做」，而是證明：

- 以 Ontology 為核心的世界模型是有價值的
- 多 Agent + 主持人 Agent 的協作流程，能比單一聊天 AI 提供更高品質的工作結果
- 輸出若能結構化成報告與行動項，會大幅提升實用性
- 即使先從個人工具台開始，整體架構也能為未來團隊協作與平台化保留空間

---

## 3. V1 核心目標

V1 的核心目標不是功能數量最大化，而是**價值驗證最大化**。

### 3.1 主要目標

1. **驗證複雜商業問題的多 Agent 收斂工作流是否真的優於單一聊天式 AI**
2. **驗證主持人 Agent 是否能夠有效提升流程品質、收斂能力與輸出結構**
3. **驗證基礎 Ontology 是否能幫助系統更穩定理解任務情境與輸出結果**
4. **驗證使用者是否會願意把系統產出直接拿來作為提案、研究、診斷或決策前置材料**
5. **驗證這套系統是否足以成為顧問服務的內部主力工具**

### 3.2 次要目標

1. 支援少量高頻單點功能型任務，提升使用黏著度
2. 為未來 SaaS 化與 Agent Registry 保留設計空間
3. 形成後續由 Codex 接手實作的穩定規格基礎

---

## 4. V1 核心使用情境

V1 應優先支援以下情境：

### 4.1 複雜商業問題分析與收斂（主場景）

例如：
- 客戶需求分析
- 專案策略方向規劃
- 營運問題診斷
- 新服務 / 新市場評估
- 提案前的多視角研究與收斂
- 商業決策前的方案比較

這類情境的共通點是：
- 問題不只一個面向
- 需要多角色、多視角分析
- 需要主持與收斂
- 最終需要輸出具結構的報告、建議與行動項

> V1 在主場景上**刻意保留彈性**，不再額外收斂成單一子類型（例如只做提案策略或只做營運診斷），以避免過早限制產品的應用範圍。

### 4.2 單點 Specialist 任務（輔助場景）

V1 先聚焦以下三類高頻單點任務：
- **合約審閱**
- **研究整理**
- **文件重組 / 提案重組**

這些任務在 V1 中應被視為：
- 補強實用性
- 提升使用頻率
- 驗證 Specialist Agent 設計
- 為未來更多單點能力擴充建立模式

但不應喧賓奪主，取代整個 V1 的核心敘事。

## 5. V1 核心功能範圍（In Scope）

### 5.1 任務輸入與工作啟動

使用者可以輸入：
- 一個複雜商業問題
- 一個研究 / 提案 / 診斷任務
- 一份文件 / 需求內容，作為任務背景

系統需能根據輸入，識別：
- 任務類型
- 問題範圍
- 是否適合啟動多 Agent 流程
- 是否適合改用單點 Specialist 流程

### 5.2 Host Agent（主持人 Agent）流程治理

V1 必須具備一個明確的主持人 Agent，其職責至少包括：
- 解析任務
- 決定流程模式
- 選擇應出場的 Agent
- 控制分析順序
- 整理觀點衝突
- 形成結論框架
- 輸出報告與行動項

### 5.3 基礎 Multi-Agent 協作流程

V1 應至少支援少量核心 Agent 的協作，並將數量控制在 **3~5 個** 之間，例如：
- Strategy / Business Analysis
- Operations
- Marketing / Market Insight
- Risk / Challenge / Devil’s Advocate

> 註：實際 Agent 名稱與類型可於後續 `05_agent_architecture_v1.md` 再細化。

V1 不追求 Agent 數量多，而追求：
- 視角差異清楚
- 功能分工合理
- 輸出可比較
- 收斂結果有價值
- 後續可持續擴充

### 5.4 基礎 Ontology 載入與使用

V1 應建立一套最小可行的 Core Ontology，使系統至少能理解：
- 任務
- 客戶 / 對象
- 問題
- 目標
- 風險
- 建議
- 行動項
- 文件 / 依據

V1 的 Ontology 不需要過度龐大，但必須足夠支撐：
- 多 Agent 在同一語境中工作
- 主持人 Agent 能有一致的收斂依據
- 結果能被結構化輸出

### 5.5 結構化輸出

V1 的輸出不能只是一段自由文字，而應至少包含：
- 問題定義
- 背景摘要
- 主要觀點 / 分析結論
- 風險與假設
- 建議方向
- 行動項 / 後續建議

視情況可加入：
- 方案比較
- 待補資料
- 優先順序

### 5.6 單點 Specialist 任務支援

V1 可支援少量單點功能型流程，例如：
- 文件審閱
- 研究整理
- 內容重組

這些任務應盡量復用：
- 相同的任務輸入框架
- 相同的結果輸出邏輯
- 相同的模型路由能力

而不是做成完全獨立、完全不同的產品。

### 5.7 LLM Router 基礎抽象層

V1 應保留模型供應商可切換的基礎設計，但不需要一開始接很多家。

V1 目標應為：
- 有統一的模型呼叫抽象層
- 至少支援 1~2 家供應商的可替換能力
- 後續可擴充更多模型

### 5.8 個人工具台介面

V1 介面應優先設計為：
- 單一使用者使用
- 任務輸入簡單直接
- 可查看分析流程與最終輸出
- **可保存任務結果與歷史**

但資料結構與系統架構上應保留未來升級為：
- 多專案
- 多工作區
- 多使用者
- 小型團隊協作

### 5.9 檔案上傳與背景資料引用

V1 應支援將檔案、文件內容或任務背景資料作為任務輸入的一部分，至少包含：
- 檔案上傳
- 任務中引用背景文件
- 將文件作為分析、審閱、研究與收斂的依據

這項能力的重要性在於：
- 顧問工作高度依賴文件與上下文
- 單點任務（如合約審閱）本身就需要原始文件
- 複合型任務也常需要簡報、需求書、合約、企劃、研究資料作為背景
- 若缺少真實背景資料，Ontology 很容易停留在抽象框架，而無法真正對應現場任務

因此，檔案上傳與背景資料引用不只是便利功能，而是 V1 能否接住真實工作流的基礎能力之一。

### 5.10 檔案 / 雲端資料來源作為知識入口

除了「把檔案附加到任務」之外，V1 在概念上應明確把這些來源視為：

> **系統知識來源的入口層，而不是單純附件功能。**

也就是說：
- 上傳的檔案
- 後續可能接入的雲端來源（例如 Google Drive 類型來源）
- 任務背景資料
- 歷史文件與專案資料

都應被理解為未來知識庫能力的原始來源。

V1 不需要一開始就做成完整知識平台，但架構上應保留這個方向：
- 檔案 / 雲端來源 → 形成 Evidence 與背景知識
- 背景知識 → 支撐任務分析
- 分析結果 → 回寫成可重用的任務成果與知識沉澱

## 6. V1 非目標（Out of Scope）

以下內容在 V1 階段**刻意不做或不完整做**：

### 6.1 完整第三方 Agent Market

V1 不開放外部自由上架 Agent。

原因：
- 治理過重
- 安全與品質控制太複雜
- 會過早把重點從核心價值驗證拉去平台營運

### 6.2 全產業完整 Ontology

V1 不追求建立適用所有產業的龐大 Ontology。

原因：
- 會過度抽象
- 難以真正落地
- 早期應優先驗證最小可行 Core Ontology

### 6.3 完整團隊協作與企業治理功能

V1 不優先處理：
- 複雜權限系統
- 多角色審批流
- 大型組織治理需求
- 完整審計追蹤

原因：
- 目前主體是個人工具台
- 應先驗證使用價值，再逐步升級

### 6.4 全自動商業動作執行

V1 不做高風險自動執行，例如：
- 自動對外發送正式文件
- 自動下達關鍵商業決策
- 自動簽核或自動承諾

原因：
- 高風險場景仍需人工作為最後核准者
- 應先聚焦分析與輸出價值

### 6.5 過度複雜的長期記憶與自主演化系統

V1 不應投入太多時間在：
- 自動自我改寫 Agent
- 自動繁殖 Agent
- 高度自治的長期記憶網絡

原因：
- 容易偏離 MVP 目標
- 不利於穩定性與可控性

### 6.6 全面替代單點專業工具

V1 不追求一開始就做成：
- 最強法務工具
- 最強研究工具
- 最強會議工具
- 最強標案工具

因為 V1 的重點不是在所有單點任務都成為最佳工具，而是先建立能夠整合與承載這些能力的平台骨架。

### 6.7 完整雲端知識庫 / 連接器平台

V1 不追求一開始就做成：
- 完整 Google Drive / Notion / 各種雲端來源同步系統
- 超複雜知識庫索引平台
- 完整跨專案全域搜尋

V1 更重要的是先證明：
- 檔案與背景資料能進入主流程
- 能形成 Evidence
- 能支撐分析與輸出
- 能沉澱為後續知識基礎

## 7. V1 成功標準

### 7.1 使用價值層

V1 若要被視為成功，至少應達成：
- 使用者願意反覆使用它處理真實顧問工作
- 使用者認為它在複雜問題收斂上明顯優於一般聊天 AI
- 使用者願意將輸出作為提案、研究、分析、診斷或內部討論基礎
- 使用者願意將其作為個人主力工具持續保存與回顧任務歷史

### 7.2 產品層

- Host Agent 流程清楚且穩定
- 多 Agent 輸出確實有視角差異
- Ontology 能支撐基本任務建模
- 輸出結構清楚，不流於一段長文字
- 單點 Specialist 任務流程可被同一平台承載
- 檔案上傳與背景資料引用可實際支援真實任務
- 檔案 / 背景資料可作為未來知識入口而不是一次性附件

### 7.3 技術層

- 系統具備基本模型抽象層
- 可支援未來擴充更多 Agent
- 可支援未來升級為團隊型產品
- 不因 V1 的快速落地而把後續架構鎖死

---

## 8. V1 的設計取捨

V1 的核心原則不是「功能越多越好」，而是：

### 8.1 先做高價值主流程，不追求全能
把最重要的使用價值集中在：
- 複雜任務
- 多 Agent 收斂
- 結構化輸出

### 8.2 先做最小可行 Ontology，不追求理論完美
先支撐真實工作，再逐步擴張。

### 8.3 先做少量高品質 Agent，不追求 Agent 數量
重點是差異化與可收斂，而不是熱鬧。

### 8.4 先做個人工具台，不急著做大型企業版本
先讓產品在真實工作中被高頻使用，再做複雜治理。

### 8.5 先做可擴充骨架，不急著做平台營運機制
先證明底座價值，再開放更多生態層功能。

---

## 9. V1 與未來版本的關係

V1 應被視為：
- 未來 SaaS 化的前身
- 未來 Agent Registry 的基礎
- 未來團隊工作台的雛形
- 未來多產業知識包與工作流擴充的起點
- 未來知識來源層與知識庫能力的第一個入口

這代表 V1 要做到兩件事同時成立：

1. **足夠小，能真的做出來**
2. **足夠正，未來不用整個打掉重做**

---

## 10. 濃縮版範圍定義

### V1 要做
- 顧問服務業內部使用為主的個人工具台
- 複雜商業問題的多 Agent 收斂流程
- Host Agent 主導的任務治理
- 最小可行 Core Ontology
- 結構化報告與行動項輸出
- 三類單點 Specialist 任務：合約審閱、研究整理、文件 / 提案重組
- 基礎 LLM Router 抽象層
- 任務歷史保存
- 檔案上傳與背景資料引用
- 將檔案 / 背景資料作為未來知識入口的基礎設計

### V1 不做
- 完整 Agent Market
- 完整企業治理與團隊系統
- 全產業完整 Ontology
- 高風險自動執行
- 過度複雜的自治與長期記憶系統
- 所有單點工具的全面替代
- 完整雲端知識連接器與超複雜知識庫平台

---

## 11. 已確認事項

1. **V1 的複雜商業問題主場景先保留彈性，不再過早收斂成單一子場景**
2. **V1 的單點 Specialist 任務先圈定為三類**
   - 合約審閱
   - 研究整理
   - 文件重組 / 提案重組
3. **V1 的核心 Agent 先限定在 3~5 個**
4. **V1 需要任務歷史保存**
5. **V1 極度需要檔案上傳 / 背景引用作為真實工作流基礎**
6. **檔案 / 雲端來源在概念上被視為知識入口層，而不只是附件功能**

---

## 12. 結語

V1 的使命不是把最終平台一次做完，而是用最小可行、但方向正確的方式，證明這個產品的核心價值：

- 多 Agent 收斂是否真的有價值
- Host Agent 是否真的能提升品質
- Ontology 是否真的能支撐更穩定的工作流
- 結構化輸出是否真的能提升工作採用率
- 這整套系統是否真的能成為顧問服務的內部主力工具
- 檔案 / 背景來源是否能真正成為未來知識沉澱的入口

若 V1 驗證成功，後續的 SaaS 化、平台化、Agent Registry、更多產業包、更多知識來源整合與工具串接，才有穩固基礎。
