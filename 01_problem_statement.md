## 1. 文件目的

本文件的目的不是描述產品功能，而是回答以下問題：

1. **這個產品到底要解決什麼問題？**
2. **這些問題為什麼重要？**
3. **為什麼現有工具不足以解決？**
4. **為什麼需要一套以 Ontology、Multi-Agent、Host Agent 為核心的系統？**

本文件將作為後續 Ontology 設計、Agent 架構、系統設計與 Codex 實作的重要依據。

## 2. 問題總述（Problem Summary）

本產品的出發點，不只是解決一個明確、單一、狹窄的痛點，而是希望建立一個**可以持續放大價值、持續承載更多能力、持續延伸到更多工作場景**的智能工作底座。

也就是說，AI Advisory OS 不只是回答：
- 現在最痛的是什麼？

它更想回答：
- **什麼樣的系統，能讓知識型工作與顧問型工作在未來持續變得更強？**
- **什麼樣的系統，不只解決單一問題，而是能承載越來越多問題的解法？**
- **什麼樣的系統，可以在不同任務、不同產業、不同專業之間持續擴張，而不需要每次重做一套工具？**

因此，本文件的問題定義不應只停留在「哪裡很痛」，而應同時涵蓋三個層次：

1. **現場層問題**：現有知識型工作流效率低、依賴人工、難以收斂
2. **結構層問題**：現有工具彼此割裂，缺乏共享世界模型與可組合能力
3. **價值層問題**：缺乏一個能持續擴張、持續進化、持續放大價值的智能平台底座

現代知識型工作與商業決策工作，面對的已不只是「資訊不足」，而是：

- 資訊太多，且分散在不同來源
- 問題本身高度複雜，往往牽涉多角色、多專業、多步驟判斷
- 真正有價值的不是單次回答，而是可被採用的結論、建議與行動方案
- 現有 AI 工具多半偏向問答、摘要、搜尋或單一功能執行，難以穩定處理複合型商業問題
- 多數團隊仍依賴人工會議、人工整理與人工收斂，效率低且難以複製
- 即使單點工具很多，也缺少一個能把能力不斷累積、編排與擴張的核心平台

因此，真正的核心問題不是「能不能找到答案」，而是：

> **能不能建立一個可持續理解情境、調用不同專業能力、收斂多方觀點、產出具執行價值結果，並且可以持續擴張其能力邊界的智能系統。**

## 3. 現實工作中的核心痛點

### 3.1 複雜問題需要多視角，但現有流程過度依賴人工

在顧問、策略、研究、提案、法務審閱、營運診斷、標案處理等場景中，問題通常不是單一角度可以解決的。

例如：
- 一個市場進入策略要同時看市場、品牌、財務、法規、執行難度
- 一份提案要同時兼顧客戶需求、競品情報、資源配置與可行性
- 一份合約審閱可能涉及法務風險、商業條件、權利義務、執行風險
- 一個營運問題可能同時牽涉流程、組織、人員、成本與 KPI

但在現實中，這些工作通常需要：
- 大量會議
- 多人反覆溝通
- 不同部門或角色來回補充
- 人工整理結論
- 再由某個人收斂成報告或建議

這導致：
- 耗時高
- 成本高
- 品質受個人能力影響很大
- 難以複製與標準化
- 知識不容易沉澱

### 3.2 資訊很多，但缺少共享世界模型

現有團隊常常不是沒有資訊，而是：
- 文件很多
- 資料很多
- 報表很多
- 會議紀錄很多
- 外部資料也很多

問題在於，這些資訊通常分散在不同地方，且彼此缺乏統一結構。

結果是：
- 每個人理解的問題範圍不同
- 不同部門對同一個名詞有不同定義
- 資訊雖可被檢索，但難以形成一致判斷基礎
- 討論容易流於片段化與經驗化

缺少共享世界模型時，AI 很容易只能做「文件摘要機」或「問答機」，而無法真正理解這個工作世界中的物件、關係、狀態與決策脈絡。

### 3.3 現有 AI 工具大多只能處理局部任務，難以完成高品質收斂

目前市場上很多 AI 工具很強，但多半偏向以下幾類：

- 聊天問答
- 文件摘要
- 搜尋與檢索
- 單一步驟生成
- 單一功能型自動化

這些工具可以提升部分效率，但面對真實複合問題時，常會遇到以下限制：

- 缺乏跨角色協作的結構
- 缺乏多視角辯證
- 缺乏共享情境模型
- 缺乏收斂機制
- 缺乏可執行輸出格式
- 缺乏與後續行動追蹤的連結

也就是說，很多工具能幫你「做一段」，但很難幫你「做完一件真正有價值的事」。

### 3.4 單一 AI 容易過度集中於單一視角

即使單一大型模型很強，它本質上仍傾向在同一個推理軌道上完成回答。

這容易出現幾種問題：
- 回答看似完整，但其實未經真正的多方檢驗
- 難以清楚區分不同專業視角的差異
- 容易跳過反方觀點或風險檢查
- 結果可能流於一種「表面完整，但未經辯證」的答案

對於高價值的商業問題而言，真正重要的不是說得流暢，而是：
- 是否有明確依據
- 是否考慮不同立場
- 是否處理衝突與不確定性
- 是否能夠收斂出可採用的方案

### 3.5 工作成果常停留在文字輸出，無法自然延伸到後續行動

很多 AI 工具最終輸出仍是一段文字、一份摘要、或一份初稿。

但真實工作往往還需要：
- 下一步要做什麼
- 哪些事項應優先
- 誰負責
- 要追蹤哪些風險
- 是否需要補資料
- 是否需要對接外部工具

如果系統只能停在「回答問題」，就很難真正成為高價值工作平台。

### 3.6 很多團隊需要的不只是複雜決策系統，也需要高品質單點能力

另一個常被忽略的問題是：

團隊雖然會遇到複雜商業問題，但也經常需要大量高頻的單點任務，例如：
- 合約生成與審閱
- 文件重組
- 市調整理
- 設計前期研究
- 標案需求拆解
- 企劃初稿整理

如果一個平台只能處理大型複雜問題，卻無法支援高頻小任務，使用黏著度可能不足。

因此，本產品要解決的問題，不只是複雜問題的處理效率，也包括：

> **如何讓同一套核心架構，同時承載高複雜度決策流程與高頻單點任務能力。**

## 4. 為什麼現有工具不足

### 4.1 傳統顧問工作流的限制

傳統顧問、研究與策略工作流雖然成熟，但存在幾個明顯限制：
- 仰賴人的能力與經驗
- 執行成本高
- 新人難以快速複製高品質產出
- 知識沉澱零散
- 方法論常存在於少數人腦中
- 多專案並行時容易失控

### 4.2 純聊天型 AI 的限制

純聊天型 AI 擅長：
- 快速回應
- 文字生成
- 靈活對話
- 泛化知識整合

但在高價值工作場景中，其限制包括：
- 缺乏穩定的結構化流程
- 缺乏共享世界模型
- 缺乏可管理的多角色協作機制
- 缺乏可追溯的輸出治理
- 缺乏明確的後續任務物件化能力

### 4.3 純知識庫 / 檢索系統的限制

知識庫與 RAG 系統能解決「找資料」問題，但通常不等於能解決「做判斷」問題。

它們通常可以：
- 幫你找到相關文件
- 引用內容
- 做基本摘要

但未必能：
- 統一情境
- 模擬不同專業視角
- 處理衝突與取捨
- 收斂成決策建議
- 轉成後續任務與行動項

### 4.4 單一功能工具的限制

市場上存在大量優秀的專用 AI 工具，例如：
- 法務審閱工具
- 會議紀錄工具
- 研究工具
- 報告生成工具
- 內容企劃工具

這些工具可以在特定任務上非常有效，但它們通常無法自然組合成一套可跨場景、跨任務、跨專業的整體系統。

本產品要處理的不是單一功能的優化，而是：

> **如何建立一個能同時容納多種任務能力，並在必要時將它們編排成更高層次工作流的平台。**

## 5. 為什麼需要 Ontology + Multi-Agent + Host Agent

### 5.1 Ontology：解決共享世界模型問題

Ontology 的角色不是增加抽象概念，而是讓系統能夠用統一方式理解：
- 這個世界中有哪些重要物件
- 這些物件之間是什麼關係
- 目前狀態是什麼
- 哪些事件、風險、決策、任務與它們相關

沒有 Ontology，系統容易停留在「讀到什麼就回答什麼」；
有 Ontology，系統才可能逐步進入「理解這個問題所處的工作世界」。

### 5.2 Multi-Agent：解決多視角分析問題

多 Agent 並不是為了製造熱鬧對話，而是為了讓系統能夠：
- 在同一個世界模型上承載不同能力
- 從不同專業框架切入問題
- 讓分析更接近真實工作中的多角色討論
- 形成風險對照、方案比較、反方檢查與專業補充

### 5.3 Host Agent：解決流程治理與結果收斂問題

如果只有多 Agent 而沒有主持人，系統很容易變成：
- 發散討論
- 觀點重複
- 缺乏優先順序
- 無法得出最終結論

Host Agent 的角色是：
- 理解任務
- 選擇與召喚適合的 Agent
- 控制討論節奏
- 要求補證據
- 處理矛盾
- 收斂為結構化輸出
- 視情況串接後續行動

### 5.4 可切換 LLM：解決供應商綁定與任務差異問題

不同模型有不同優勢。
有些任務重視：
- 推理品質
- 成本
- 速度
- 長上下文
- 工具使用能力
- 輸出穩定性

若系統綁死在單一供應商，長期彈性會受限。
因此，平台層需保有模型切換與路由能力，讓系統可以根據任務選擇更適合的模型。

### 5.5 Structured Report and Action Output：解決「只會回答，不會落地」的問題

真正有價值的輸出不應只是一段對話，而應盡量物件化、結構化，並能往後延伸。例如：
- 問題定義
- 事實與依據
- 分析結論
- 風險與假設
- 方案比較
- 建議採用方向
- 後續行動項
- 追蹤事項

這樣系統才有機會從「AI 助理」進化成「智能工作平台」。

## 6. 為什麼先從顧問服務業切入

本產品長期不是只服務顧問，但第一版從顧問服務業開始驗證，有以下幾個理由：

### 6.1 顧問工作本身就是高密度複合型知識工作

顧問工作天然包含：
- 研究
- 分析
- 提案
- 診斷
- 報告輸出
- 多視角整合
- 客製化判斷

這與本產品的核心能力高度重疊。

### 6.2 創建者本身最容易提供真實、高頻、可持續的使用場景

若產品第一版直接瞄準過度寬泛市場，會因為問題類型過多而無法收斂。
相反地，從創建者自己高頻使用的場景開始，有幾個好處：
- 回饋更快
- 問題更真實
- 更容易反覆修正
- 更容易驗證哪些能力真正有價值

### 6.3 顧問場景可自然外溢到其他知識型工作

顧問服務業的很多能力，其實可延伸到：
- 公關公司
- 市調研究
- 線上教育內容規劃
- 設計前期調研
- 標案與提案工作
- 企業內部策略專案

也就是說，顧問場景既具代表性，又具延展性。

### 6.4 顧問工作最能驗證主持人 Agent 與多 Agent 協作價值

如果只是單點功能型場景，很多時候用單一 Specialist Agent 就足夠。
但顧問工作更需要：
- 任務拆解
- 多視角分析
- 衝突處理
- 方案比較
- 邏輯收斂
- 最終提案與行動建議

因此，顧問場景特別適合驗證本產品最核心的差異化能力。

## 7. 本產品真正要解決的核心矛盾

若將所有問題濃縮，本產品真正要解決的核心矛盾是：

### 7.1 知識型工作需要更高階的智能協作，但現有工具多停留在局部效率提升

也就是說：
- 工具有很多
- 功能越來越強
- AI 越來越會說

但真正的工作價值仍經常卡在：
- 情境理解不足
- 多視角整合不足
- 決策收斂不足
- 輸出與行動脫節
- 平台擴充性不足

### 7.2 團隊需要的是「能被持續擴充的智能工作底座」而不是一堆分散工具

本產品的目標不是再做一個單點 AI 工具，而是建立一個底座，讓未來可以不斷往上疊加：
- 新 Agent
- 新場景
- 新任務
- 新產業包
- 新工具串接
- 新工作流

### 7.3 使用者需要的不是更長的回答，而是更可採用的結果

也就是從：
- 會回答
- 會生成
- 會摘要

走向：
- 會理解
- 會辯證
- 會收斂
- 會形成建議
- 會延伸成行動

### 7.4 真正高價值的系統，不只是解決今天的問題，而是能承載明天更多問題

這一點是本產品與一般單點工具最大的差異。

很多工具的邏輯是：
- 先解一個問題
- 再做一個新工具解下一個問題
- 工具越堆越多，知識與流程越分散

本產品希望反過來：
- 建立一個可持續擴張的核心系統
- 讓新的能力、方法論、角色與場景，可以疊加在同一個底座上
- 讓每一次新增能力，都能讓整個平台更有價值，而不是更碎片化

這也代表，本產品追求的不只是效率提升，而是：

> **價值的累積性、能力的可擴張性、以及工作成果的可複製性。**

---

## 8. 問題定義（Condensed Statement）

AI Advisory OS 要解決的核心問題可以濃縮如下：

> 在複雜、跨專業、跨資訊來源的知識型工作中，現有工具雖能提升局部效率，卻難以建立共享世界模型、承載多視角分析、收斂出可執行結果，並同時兼容高頻單點任務與長期平台擴充需求。
>
> 更進一步來說，市場上缺少的不是更多零散功能，而是一個能夠持續累積能力、持續延伸場景、持續放大價值的智能工作底座。
>
> 因此，需要一套以 Ontology 為核心、以可擴充 Agent 能力為載體、以 Host Agent 為治理中心、並可輸出結構化報告與行動方案的智能工作平台。

---

## 9. 本文件對後續設計的影響

根據本文件，後續設計應遵循以下方向：

1. **Ontology 設計必須優先於大量 prompt 設計**
2. **Agent 設計要區分複合型協作與單點 Specialist 能力**
3. **Host Agent 必須是系統核心，而不是附加功能**
4. **輸出格式需結構化，不能只停留在對話文字**
5. **V1 應先從顧問服務業的高頻真實任務切入**
6. **系統架構需預留長期擴充性，不可做成一次性專案工具**

## 11. 結語

本產品不是因為市場上已經有很多 AI 工具，所以再多做一個 AI 工具；而是因為現有工具雖能提升局部效率，卻仍無法穩定承載高價值知識工作所需的：

- 共享情境理解
- 多視角能力調用
- 邏輯治理與結果收斂
- 單點能力與複合工作流兼容
- 結果落地與後續追蹤
- 長期平台化擴充能力

更關鍵的是，本產品不是只想解決今天的一個具體痛點，而是想打造一個：

- 能持續擴張能力
- 能持續累積方法論
- 能持續延伸到更多工作場景
